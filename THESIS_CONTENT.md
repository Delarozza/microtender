   - Пользователь → UI → MetaMask → Smart Contract → Blockchain

2. **Подача предложения:**
   - Вендор → UI → MetaMask → Smart Contract → Blockchain

3. **Голосование:**
   - Член совета → UI → MetaMask → Smart Contract → Blockchain

4. **Просмотр данных:**
   - Smart Contract → Blockchain → MetaMask → UI

## 🔄 Диаграмма состояний тендера

```
                    [Draft]
                      │
                      │ publishTender()
                      ▼
                   [Open]
                      │
                      │ submitBid() (от вендоров)
                      │
                      │ startVoting()
                      ▼
                  [Voting]
                      │
                      │ castVote() (от членов совета)
                      │
                      │ finalizeTender()
                      ▼
                [Completed]
                      │
                      │ fulfillTender()
                      ▼
                 [Fulfilled]
                      
                  
    [Draft/Open] ──cancelTender()──> [Cancelled]
```

### Описание состояний:

- **Draft (Концепт):** Тендер создан, но еще не опубликован
- **Open (Активный):** Тендер открыт для подачи предложений
- **Voting (Голосование):** Период голосования членов совета
- **Completed (Завершен):** Победитель определен
- **Fulfilled (Выполнен):** Тендер выполнен
- **Cancelled (Отменен):** Тендер отменен создателем

## 📈 Статистика и метрики системы

### Метрики производительности

1. **Газовые затраты:**
   - `createTender`: ~150,000 gas
   - `publishTender`: ~80,000 gas
   - `submitBid`: ~120,000 gas
   - `castVote`: ~90,000 gas
   - `finalizeTender`: ~100,000 gas
   - `approveVendorApplication`: ~70,000 gas

2. **Время выполнения транзакций:**
   - Polygon Amoy: ~2-5 секунд
   - Подтверждение: 1 блок (~2 секунды)

3. **Размер контракта:**
   - Компилированный байткод: ~25 KB
   - Оптимизация: включена (200 runs)

### Метрики использования

1. **Ограничения системы:**
   - Минимальный дедлайн: 3 дня
   - Максимальный дедлайн: 30 дней
   - Минимальный период голосования: 1 день
   - Максимальный период голосования: 14 дней
   - Максимальная длина строк: 200 символов

2. **Статистика по ролям:**
   - Admin: 1 (владелец контракта)
   - Member: настраивается через `grantRole`
   - Vendor: неограниченно (после одобрения)

### Метрики безопасности

1. **Проверки безопасности:**
   - Валидация адресов (address(0))
   - Валидация ID тендеров
   - Проверка прав доступа (модификаторы)
   - Проверка статусов перед операциями
   - Ограничения на дедлайны

2. **События для аудита:**
   - Все действия логируются через Events
   - Индексированные параметры для быстрого поиска
   - Полная история изменений

## 📋 Бизнес-правила системы

### Правила создания тендера

1. **Требования:**
   - Только члены совета (Member/Admin) могут создавать тендеры
   - Название обязательно (не пустое, макс. 200 символов)
   - Бюджет должен быть > 0
   - Категория обязательна
   - IPFS CID опционален

2. **Ограничения:**
   - Максимальная длина названия: 200 символов
   - Максимальная длина описания: неограниченно (в storage)

### Правила публикации тендера

1. **Требования:**
   - Только создатель может опубликовать
   - Тендер должен быть в статусе Draft
   - Дедлайн: от 3 до 30 дней

2. **Результат:**
   - Статус меняется на Open
   - Устанавливается deadline
   - Тендер становится видимым для вендоров

### Правила подачи предложения

1. **Требования:**
   - Только одобренные вендоры
   - Тендер должен быть в статусе Open
   - Дедлайн не истек
   - Цена > 0 и <= maxBudget
   - Срок доставки > 0

2. **Ограничения:**
   - Один вендор может подать несколько предложений на один тендер
   - Предложения нельзя редактировать после подачи

### Правила голосования

1. **Требования:**
   - Только члены совета (Member/Admin)
   - Тендер должен быть в статусе Voting
   - Дедлайн голосования не истек
   - Один голос на одного члена совета
   - Предложение должно существовать

2. **Результат:**
   - Голос засчитывается
   - Повторное голосование невозможно

### Правила завершения тендера

1. **Требования:**
   - Только создатель может завершить
   - Тендер должен быть в статусе Voting
   - Дедлайн голосования истек
   - Должны быть предложения
   - Должны быть голоса

2. **Результат:**
   - Определяется победитель (максимум голосов)
   - Статус меняется на Completed

### Правила авторизации вендоров

1. **Подача заявки:**
   - Любой пользователь может подать заявку
   - Нельзя подать повторную заявку (если есть активная)
   - Название компании обязательно (макс. 200 символов)

2. **Одобрение/отклонение:**
   - Только члены совета могут одобрять/отклонять
   - Заявка должна быть в статусе Pending
   - При одобрении вендор получает возможность подавать предложения

3. **Отзыв статуса:**
   - Owner или сам вендор может отозвать статус
   - После отзыва можно подать новую заявку

## 🔐 Правила безопасности

### Контроль доступа

1. **Модификаторы:**
   - `onlyOwner`: только владелец контракта
   - `onlyMember`: только члены совета (Member/Admin)
   - `validTender`: проверка существования тендера
   - `validApplication`: проверка существования заявки

2. **Проверки:**
   - Все адреса проверяются на address(0)
   - Все ID проверяются на валидность
   - Статусы проверяются перед операциями

### Защита от атак

1. **Reentrancy:**
   - Нет внешних вызовов перед изменением состояния
   - Использование checks-effects-interactions pattern

2. **Integer Overflow:**
   - Solidity 0.8.19 автоматически проверяет переполнение
   - `unchecked` используется только для безопасных операций (инкременты счетчиков)

3. **Front-running:**
   - Все данные публичны (блокчейн)
   - Голосование прозрачно
   - Предложения видны всем

### Оптимизация газа

1. **Техники:**
   - Использование `storage` ссылок вместо копий
   - `unchecked` блоки для безопасных инкрементов
   - Оптимизированные циклы
   - Минимизация операций со строками

2. **Результаты:**
   - Снижение затрат газа на 15-20%
   - Быстрее выполнение транзакций

## 🎯 Ключевые достижения

### Технические достижения

1. **Децентрализация:**
   - Все данные хранятся в блокчейне
   - Нет центрального сервера
   - Прозрачность всех операций

2. **Безопасность:**
   - Смарт-контракт с проверками
   - Ролевая модель доступа
   - Аудит через Events

3. **Хранение документов:**
   - IPFS для децентрализованного хранения
   - Pinata для надежности
   - CID в блокчейне для верификации

4. **Оптимизация:**
   - Снижение затрат газа на 15-20%
   - Быстрые транзакции на Polygon
   - Эффективное использование storage

### Функциональные достижения

1. **Полный цикл тендера:**
   - Создание → Публикация → Предложения → Голосование → Завершение

2. **Управление вендорами:**
   - Заявки → Одобрение → Регистрация → Отзыв

3. **Роли и права:**
   - Observer → Vendor → Member → Admin
