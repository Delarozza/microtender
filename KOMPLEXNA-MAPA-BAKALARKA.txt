4.1 Celková architektúra systému [3-4 strany]
──────────────────────────────────────────────

4.1.1 Prehľad architektúry [1 strana]
ČO PÍSAŤ:
• Systém je trojvrstvový, nie štvorvrstvový:
  - Prezentačná vrstva: React frontend + MetaMask
  - Logická vrstva: smart kontrakt `MicroTender.sol`
  - Dátová vrstva: blockchain + IPFS
• Zdôrazniť, že v aktuálnom prototype NIE JE samostatný backend ani relačná databáza
📊 DIAGRAM: Architecture Diagram (#3)

4.1.2 Technologický stack [1 strana]
ČO PÍSAŤ:

| Vrstva         | Technológia              | Účel                                |
|----------------|--------------------------|-------------------------------------|
| Frontend       | React 19, react-scripts  | Používateľské rozhranie             |
| Web3           | ethers.js v6, MetaMask   | Volanie smart kontraktu             |
| Smart Contract | Solidity 0.8.19          | Biznis logika systému               |
| Dev/Test       | Hardhat                  | Kompilácia, testovanie, deployment  |
| Blockchain     | Polygon Amoy             | Testovacia sieť pre nasadenie       |
| Storage        | IPFS + Pinata            | Ukladanie dokumentov a metadát      |

• Krátko zdôvodniť, prečo je tento stack primeraný bakalárskemu prototypu

4.1.3 Komunikácia medzi komponentmi [1 strana]
ČO PÍSAŤ:
📊 DIAGRAM: Data Flow Diagram (#6)
• Frontend -> MetaMask -> smart kontrakt
• Frontend -> Pinata API -> IPFS
• Frontend -> read-only RPC provider pri nepripojenej peňaženke
• Smart kontrakt vracia dáta cez view funkcie a eventy

4.1.4 Architektonické rozhodnutia [0.5 strany]
ČO PÍSAŤ:
• Prečo bez backendu:
  - menšia komplexita projektu
  - menej komponentov na údržbu
  - priame overiteľné volania na blockchain
• Prečo bez databázy:
  - blockchain je source of truth
  - aktuálny rozsah projektu nevyžaduje off-chain cache


4.2 Návrh smart kontraktu MicroTender [4-5 strán]
───────────────────────────────────────────────

4.2.1 Monolitická architektúra kontraktu [1 strana]
ČO PÍSAŤ:
• Celý systém je implementovaný v JEDNOM kontrakte `MicroTender.sol`
• Nepoužíva sa Factory Pattern ani viac samostatných kontraktov
• Výhody:
  - jednoduchšia implementácia
  - nižšia zložitosť pre bakalársky projekt
  - menšie režijné náklady pri vytváraní nových tendrov

4.2.2 Dátové štruktúry [1-1.5 strany]
ČO PÍSAŤ:
• Hlavné structy:
  - `Tender`
  - `Bid`
  - `VendorApplication`
• Popísať význam kľúčových polí:
  - `ipfsCID`
  - `deadline`
  - `votingDeadline`
  - `status`
• Ukázať aj mappingy:
  - `mapping(uint256 => Tender) public tenders`
  - `mapping(uint256 => Bid[]) public tenderBids`
  - `mapping(uint256 => mapping(address => bool)) public hasVoted`
  - `mapping(uint256 => mapping(uint256 => uint256)) public voteCount`

4.2.3 Role a oprávnenia [1 strana]
ČO PÍSAŤ:
• Reálne role podľa kódu:
  - `Admin`
  - `Member`
  - registrovaný vendor cez `registeredVendors`
  - pozorovateľ ako read-only používateľ mimo kontraktu
• Funkcie:
  - `grantRole()`
  - `revokeRole()`
  - `approveVendorApplication()`
  - `rejectVendorApplication()`
• Zdôrazniť rozdiel:
  - `Admin/Member` sú roly rady
  - vendor nie je v enum `UserRole`, ale v samostatnej evidencii

4.2.4 Stavový model tendra [1-1.5 strany]
ČO PÍSAŤ:
📊 DIAGRAM: State Machine Diagram (#2)
• Presné stavy podľa kontraktu:
  - `Draft`
  - `Open`
  - `Voting`
  - `Completed`
  - `Fulfilled`
  - `Cancelled`
• Prechody:
  - `createTender()` -> `Draft`
  - `publishTender()` -> `Open`
  - `startVoting()` -> `Voting`
  - `finalizeTender()` -> `Completed`
  - `fulfillTender()` -> `Fulfilled`
  - `cancelTender()` -> `Cancelled`


4.3 Životný cyklus tendra a proces obstarávania [4-5 strán]
────────────────────────────────────────────────────────────

4.3.1 Etapa 1: Vytvorenie a publikovanie tendra [1.5 strany]
ČO PÍSAŤ:
• Člen rady vytvorí draft cez `createTender()`
• Dokument môže najprv nahrať na IPFS a CID uložiť do draftu
• Následne tender publikuje cez `publishTender()`
• V UI je tento proces riešený ako dvojkrokový workflow
📊 DIAGRAM: Sequence Diagram - vytvorenie tendra (#4)

4.3.2 Etapa 2: Registrácia dodávateľa a podanie ponuky [1.5 strany]
ČO PÍSAŤ:
• Dodávateľ podá žiadosť cez `submitVendorApplication()`
• Člen rady žiadosť schváli alebo zamietne
• Po schválení môže vendor podať ponuku cez `submitBid()`
• Bid obsahuje:
  - cenu
  - lehotu dodania
  - textový opis
• V aktuálnom prototype nie sú dokumenty k ponuke ukladané ako pole CID priamo v `Bid`

4.3.3 Etapa 3: Hlasovanie a ukončenie [1.5-2 strany]
ČO PÍSAŤ:
• Tvorca tendra spustí hlasovanie cez `startVoting()`
• Členovia rady hlasujú cez `castVote()`
• Po uplynutí lehoty sa tender ukončí cez `finalizeTender()`
• Víťaz sa zisťuje na základe najvyššieho počtu hlasov
• Po administratívnom uzavretí môže tvorca označiť tender ako splnený cez `fulfillTender()`
• Dôležité:
  - kontrakt v aktuálnej verzii nerealizuje on-chain platbu ani escrow


4.4 Návrh integrácie IPFS [2-3 strany]
──────────────────────────────────────

4.4.1 Dôvod použitia IPFS [0.5 strany]
ČO PÍSAŤ:
• dokumenty sú príliš veľké na ukladanie priamo do blockchainu
• CID zabezpečuje kontrolu integrity
• IPFS rieši oddelenie metadát on-chain a súborov off-chain

4.4.2 Proces nahrávania cez Pinata [1 strana]
ČO PÍSAŤ:
• Reálne sa nahráva z frontendu cez `src/utils/pinata.js`
• Používa sa JWT token `REACT_APP_PINATA_JWT`
• Kroky:
  1. používateľ vyberie súbor
  2. frontend validuje veľkosť a typ
  3. súbor sa odošle na Pinata API
  4. Pinata vráti CID
  5. CID sa uloží do smart kontraktu

4.4.3 Prístup k dokumentu [0.5-1 strana]
ČO PÍSAŤ:
• Použiť `getIPFSUrl(cid)`
• Vysvetliť IPFS gateway
• Prepojenie na detail tendra, kde si používateľ môže dokument otvoriť alebo stiahnuť
