## 1. Введение

### 1.1 Актуальность темы

В современном мире прозрачность и подотчетность в управлении финансами становятся критически важными, особенно для организаций, работающих с публичными средствами. Студенческие советы, как представительные органы студенческого самоуправления, регулярно проводят тендеры для закупки товаров и услуг. Однако традиционные централизованные системы управления тендерами имеют ряд недостатков:

- **Отсутствие прозрачности:** Процесс принятия решений скрыт от общественности
- **Риск манипуляций:** Централизованные системы могут быть подвержены коррупции
- **Отсутствие неизменности данных:** Данные могут быть изменены или удалены
- **Зависимость от посредников:** Необходимость доверия к третьим сторонам

Блокчейн-технология предлагает решение этих проблем, обеспечивая децентрализованное, прозрачное и неизменяемое хранение данных. Интеграция с IPFS (InterPlanetary File System) позволяет эффективно хранить документы вне блокчейна, сохраняя при этом их целостность и доступность.

### 1.2 Цель и задачи работы

**Цель работы:** Разработка децентрализованной системы управления тендерами для студенческого совета на базе блокчейна Polygon и IPFS, обеспечивающей прозрачность, безопасность и эффективность процесса проведения тендеров.

**Задачи:**
1. Провести анализ существующих решений для управления тендерами
2. Изучить теоретические основы блокчейн-технологий и IPFS
3. Спроектировать архитектуру децентрализованной системы управления тендерами
4. Разработать смарт-контракт с реализацией бизнес-логики системы
5. Реализовать пользовательский интерфейс для взаимодействия с системой
6. Интегрировать IPFS для хранения документов тендеров
7. Провести тестирование системы и анализ результатов

### 1.3 Объект и предмет исследования

## 3. Теоретические основы

### 3.1 Блокчейн технология

Блокчейн — это распределенная база данных, которая хранит информацию в виде цепочки блоков, каждый из которых содержит хеш предыдущего блока. Это обеспечивает:

- **Неизменяемость:** Данные не могут быть изменены после записи
- **Прозрачность:** Все транзакции видны всем участникам
- **Децентрализация:** Нет единой точки отказа

### 3.2 Смарт-контракты

Смарт-контракт — это самоисполняющийся код, который автоматически выполняет условия соглашения при выполнении определенных условий. В системе MicroTender смарт-контракт реализует:

- Логику создания и управления тендерами
- Систему голосования
- Управление ролями и правами доступа
- Автоматическое определение победителя

### 3.3 Polygon Network

Polygon — это решение для масштабирования Ethereum, которое обеспечивает:

- **Низкие комиссии:** Транзакции стоят доли цента
- **Быстрое подтверждение:** ~2 секунды
- **Совместимость с Ethereum:** Использует те же инструменты и стандарты

### 3.4 IPFS (InterPlanetary File System)

IPFS — это протокол для распределенного хранения файлов, который:

- Использует контентно-адресуемое хранение (CID)
- Обеспечивает децентрализованное хранение
- Гарантирует целостность данных через хеширование

В системе MicroTender IPFS используется для хранения документов тендеров (PDF, DOC, DOCX), что позволяет:

- Экономить газ (не хранить большие файлы в блокчейне)
- Обеспечивать доступность документов
- Сохранять неизменяемость через CID

## 4. Проектирование системы MicroTender

### 4.1 Архитектура системы

Система MicroTender состоит из трех основных компонентов:

1. **Frontend (React):** Пользовательский интерфейс
2. **Smart Contract (Solidity):** Бизнес-логика на блокчейне
3. **IPFS (Pinata):** Хранение документов

### 4.2 Модель данных

#### 4.2.1 Структура Tender

```solidity
struct Tender {
    uint256 id;
    address creator;
    string title;
    string description;
    uint256 maxBudget;
    string category;
    string ipfsCID;      // CID документа из IPFS
    uint256 deadline;
    uint256 votingDeadline;
    TenderStatus status;
    uint256 createdAt;
}
```

#### 4.2.2 Структура Bid

```solidity
struct Bid {
    uint256 id;
    uint256 tenderId;
    address vendor;
    uint256 price;
    uint256 deliveryTime;
    string description;
    uint256 submittedAt;
}
```

#### 4.2.3 Структура VendorApplication

```solidity
struct VendorApplication {
    uint256 id;
    address applicant;
    string companyName;
    string contactInfo;
    string description;
    uint256 submittedAt;
    ApplicationStatus status;
}
```

### 4.3 Бизнес-логика

#### 4.3.1 Жизненный цикл тендера

1. **Draft (Концепт):** Тендер создан, но не опубликован
2. **Open (Активный):** Тендер открыт для подачи предложений
3. **Voting (Голосование):** Период голосования членов совета
4. **Completed (Завершен):** Победитель определен
5. **Fulfilled (Выполнен):** Тендер выполнен
6. **Cancelled (Отменен):** Тендер отменен создателем

#### 4.3.2 Роли и права доступа

- **Observer:** Просмотр тендеров, подача заявки на регистрацию
- **Vendor:** Подача предложений (после одобрения)
- **Member:** Создание тендеров, голосование, одобрение вендоров
- **Admin:** Все права Member + управление ролями

### 4.4 Безопасность

#### 4.4.1 Модификаторы доступа

```solidity
modifier onlyOwner() {
    require(msg.sender == owner, "Only owner");
    _;
}

modifier onlyMember() {
    require(hasRole[msg.sender] && userRoles[msg.sender] >= UserRole.Member, "Only members");
    _;
}

modifier validTender(uint256 _tenderId) {
    require(_tenderId > 0 && _tenderId <= tenderCounter, "Invalid tender ID");

```

#### 4.4.2 Валидация данных

- Проверка адресов на `address(0)`
- Проверка длины строк (максимум 200 символов)
- Проверка диапазонов (дедлайны, бюджеты)
- Проверка статусов перед операциями

#### 4.4.3 Оптимизация газа

- Использование `unchecked` блоков для безопасных инкрементов
- Оптимизированные циклы
- Минимизация операций со строками
- Использование `storage` ссылок

## 5. Реализация системы

### 5.1 Разработка смарт-контракта

Смарт-контракт MicroTender реализован на Solidity версии 0.8.19 и включает:

- **Основные функции:**
  - `createTender()` - создание тендера
  - `publishTender()` - публикация тендера
  - `submitBid()` - подача предложения
  - `castVote()` - голосование
  - `finalizeTender()` - завершение тендера

- **Управление вендорами:**
  - `submitVendorApplication()` - подача заявки
  - `approveVendorApplication()` - одобрение
  - `rejectVendorApplication()` - отклонение
  - `revokeVendorStatus()` - отзыв статуса

- **Управление ролями:**
  - `grantRole()` - выдача роли
  - `revokeRole()` - отзыв роли
  - `getUserRole()` - получение роли

### 5.2 Интеграция IPFS

Для интеграции IPFS используется сервис Pinata, который предоставляет:

- API для загрузки файлов
- Гатевей для доступа к файлам
- Надежное хранение (pinning)

Процесс загрузки документа:

1. Пользователь выбирает файл в UI
2. Файл загружается в Pinata через API
3. Получается CID (Content Identifier)
4. CID сохраняется в смарт-контракте
5. Документ доступен через IPFS гатевей

### 5.3 Разработка пользовательского интерфейса

Пользовательский интерфейс реализован на React и включает:

- **Компоненты:**
  - Список тендеров с фильтрацией
  - Форма создания тендера
  - Форма подачи предложения
  - Панель голосования
  - Панель управления вендорами

- **Интеграция с блокчейном:**
  - Подключение через MetaMask
  - Использование ethers.js для взаимодействия с контрактом
  - Обработка событий блокчейна

### 5.4 Технический стек

**Frontend:**
- React 18
- Tailwind CSS
- ethers.js
- lucide-react (иконки)

**Blockchain:**
- Solidity 0.8.19
- Polygon Amoy (тестовая сеть)
- MetaMask (кошелек)

**Storage:**
- Pinata IPFS
- IPFS Protocol
